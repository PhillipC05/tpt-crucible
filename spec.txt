Software Design Document: TPT Crucible
Version: 1.0
Status: Draft / Architecture Phase
Brand Identity: Deep-tech, disruptive, physics-meets-software, democratizing hardware.
1. Executive Summary & Purpose
TPT Crucible is a unified, hardware-agnostic software suite designed to bypass traditional AI hardware monopolies. Its purpose is to allow engineers, researchers, and DIY hardware builders to compile, simulate, and deploy standard AI models onto non-traditional, custom-built hardware architectures.
Instead of forcing users to adapt their AI to fit commercial GPUs, TPT Crucible adapts the AI to fit the physical reality of custom FPGAs, analog compute-in-memory circuits, and distributed microcontroller swarms.
2. High-Level Architecture
The suite operates on a "Core and Modules" architecture. The Core translates the AI into a raw mathematical format, and the Modules translate that math into physical hardware instructions.
The Core: TPT Catalyst (Model Ingestion & Intermediate Representation)
Module 1 (FPGA): TPT Fusion (High-Bandwidth Logic Synthesis)
Module 2 (Analog): TPT Element (Physics-to-Weight Mapping & Noise Simulation)
Module 3 (Swarm): TPT Alloy (Distributed Graph Partitioning & Firmware Generation)
The Dashboard: TPT Observer (Real-time Telemetry & Hardware Monitoring)
3. Module Specifications & Data Flow
3.1 The Core: TPT Catalyst
Purpose: The universal translator. It ingests standard AI models and strips away the GPU-specific assumptions, converting them into a pure, hardware-agnostic mathematical graph.
Key Feature: Operator Fusion. It automatically combines sequential math operations (like a matrix multiplication followed by an activation function) into single, highly optimized mathematical blocks to reduce memory movement.
Key Feature: Quantization Auto-Search. Rather than requiring the user to pick a quantization scheme manually and then validate accuracy separately, Catalyst accepts an accuracy budget flag (`--accuracy-budget 0.05` = max 5% perplexity increase). It tries INT4, validates against a fast SiL pass or `.tptprofile` sensitivity data, and automatically promotes fragile layers to INT8 until the budget is met. No manual iteration required.
Key Feature: Streaming Pre-flight. Operator compatibility analysis is streamed to Observer in real time as the graph is traversed — users see pass/warn/fail badges appearing live rather than waiting for a full blocking report. Each warning carries a one-click "Apply suggested fix" action for common substitutions (Flash Attention → standard MHA, SwiGLU → GELU, RMSNorm → LayerNorm).
Key Feature: Toolchain Verifier (`tpt-doctor`). A single command that scans for all required external tools, checks versions, reports what's missing with install instructions, and runs a minimal end-to-end smoke test. Eliminates the silent broken-install problem for new users.
Supported Input Formats:
  - GGUF (.gguf) — primary; quantization-preserving; reads from Spark's model library
  - SafeTensors (.safetensors) — primary; the current HuggingFace standard for model distribution; memory-mapped, no pickle deserialization
  - HuggingFace model directory — ingest an HF model folder (config.json + model.safetensors) or pull directly from the Hub (`--hf-repo org/model-name`)
  - ONNX (.onnx) — cross-framework standard
  - PyTorch (.pt / TorchScript) — legacy and research models
  - TensorFlow SavedModel — TF2 standard
  - TFLite (.tflite) — Google's edge deployment format; models arrive pre-quantized and operator-simplified, well-matched to FPGA/swarm targets
  - AWQ / GPTQ (HuggingFace repos with quantize_config.json) — ingest preserves existing quantization metadata; Catalyst passes it directly to hardware targets rather than re-quantizing
  - EXL2 (.exl2) — ExLlamaV2 mixed-precision format; per-layer quantization metadata feeds directly into Catalyst's mixed-precision search
  - JAX / Flax orbax checkpoints — opens the academic/research model pipeline (Google Research, DeepMind)
  - Llamafile (.llamafile) — header-stripped to GGUF; handled by the existing GGUF ingestion path
  - Keras (.h5 / .keras) — routed through TF SavedModel conversion
Output: The TPT Intermediate Representation (TPT-IR), a standardized JSON/binary format representing the pure math of the neural network.
3.2 Module 1: TPT Fusion (For FPGA + HBM)
Purpose: Takes the TPT-IR and generates the hardware description (Verilog/VHDL) to build custom Matrix Multiplication (MAC) arrays on an FPGA, specifically optimized to interface with High Bandwidth Memory (HBM) controllers.
Key Feature: HBM Auto-Router. A library of pre-verified, open-source memory controllers. The user selects their FPGA board, and TPT Fusion automatically wires the compute arrays to the HBM pins, handling the complex timing constraints automatically.
Key Feature: FPGA Overlay Architecture. A pre-synthesized "AI overlay" — a fixed bitstream containing a configurable datapath (parameterized MAC arrays, weight BRAM banks, HBM controller). Model compilation writes weight data + datapath configuration into the overlay rather than triggering a full Yosys+Nextpnr resynthesis run. This reduces the per-model compile cycle from 30–120 minutes to approximately 10 seconds, making iterative development practical. Full resynthesis is still available for custom board targets not covered by a pre-built overlay.
Output: Synthesizable RTL (Register Transfer Level) code, memory initialization files, and overlay configuration files (.fusecfg) for overlay-compatible targets.
3.3 Module 2: TPT Element (For Analog Compute)
Purpose: Translates floating-point AI weights into physical electrical components (resistors, memristors, op-amp gains) and simulates the physical reality of the circuit.
Key Feature: The "Reality Check" Engine. Standard simulators assume perfect parts. TPT Element injects simulated thermal noise, voltage drift, and component tolerance errors. It tells the user exactly where the physical circuit will fail due to real-world physics and suggests hardware mitigations (e.g., "Add a heatsink to Node 4" or "Use 1% tolerance resistors here").
Output: SPICE netlists, PCB layout recommendations, and a "Confidence Score" for the analog design.
3.4 Module 3: TPT Alloy (For Microcontroller Swarms)
Purpose: Takes a massive AI model and mathematically shatters it into tiny shards, distributing them across hundreds of cheap microcontrollers (ESP32, RP2040, RISC-V) connected in a physical mesh.
Key Feature: Topology-Aware Partitioning. The user inputs the physical wiring layout of their swarm (e.g., a 2D grid or a star). TPT Alloy uses graph theory to assign neural network layers to specific physical nodes, minimizing the data travel time across the physical wires.
Key Feature: Physical Topology Auto-Discovery. As an alternative to manually describing the wiring layout, swarm nodes can measure round-trip timing to all other nodes and report a latency matrix back to a coordinator. Alloy reconstructs the physical topology automatically from that matrix, reducing first-time setup from "describe your wiring in JSON" to "power on your nodes." User confirms the inferred topology before partitioning begins.
Key Feature: Attention-Head Parallel Partitioning. For transformer models, attention heads within a single layer are embarrassingly parallel — each head is fully independent during the forward pass and requires only a sum-reduce at the end. Alloy supports a hybrid partitioning strategy: heads are distributed across nodes in parallel (one or more heads per node), then layer-serial partitioning resumes for the FFN sublayers. This is a substantial throughput improvement over pure layer-wise partitioning for transformer architectures and is selected automatically by the Topology Advisor when the model is transformer-based.
Key Feature: KV Cache Distribution. For autoregressive LLM inference, each node owns the KV cache for the attention heads assigned to it. Only the current query and key vectors are streamed between nodes — not the full growing cache — preventing out-of-memory conditions on nodes with limited PSRAM. The partition planner performs a second-pass KV allocation after layer assignment to ensure no node exceeds its memory budget.
Key Feature: Fault-Tolerant Execution. Each node emits a periodic heartbeat. If a node goes dark mid-inference (brown-out, WiFi dropout, thermal throttle), the swarm coordinator automatically reassigns its layers to the nearest available neighbors and continues in degraded mode rather than crashing. Observer surfaces a dead-node heatmap in real time. Rollback to full topology resumes automatically if the node recovers.
Key Feature: Pipeline Parallelism. While node N is processing token T at its layer range, nodes earlier in the chain start processing token T+1. This rolling pipeline eliminates the inference stall that naive sequential execution causes and delivers up to N× throughput improvement on an N-node chain. Nodes buffer a rolling window of KV state to support in-flight tokens. The firmware scheduler is configurable: pipeline depth can be set to 1 (sequential, lowest memory) up to the full node count (maximum throughput).
Output: Unique, node-specific C++/Rust firmware for every microcontroller in the swarm, plus a master flashing script and a heartbeat-enabled runtime protocol.
3.5 The Dashboard: TPT Observer
Purpose: The unified control center to monitor the physical hardware once it is built and running.
Key Feature: Hardware Agnostic Telemetry. It tracks tokens-per-second, memory bandwidth utilization (for FPGAs), thermal drift (for Analog), and node-latency (for Swarms) in a single, unified cyberpunk-industrial UI.
4. Recommended Technology Stack
To build this efficiently and maintain the "open, disruptive" ethos of the TPT brand, the stack relies heavily on high-performance languages (Rust/C++) for the heavy lifting, and modern web frameworks for the UI.
4.1 Core & AI Ingestion (TPT Catalyst)
Language: Python (Frontend/Ingestion) + Rust (Backend Compiler).
AI Frameworks: PyTorch, ONNX Runtime (for model parsing), safetensors (HuggingFace SafeTensors library), huggingface_hub (Hub API for model download + metadata).
Compiler Infrastructure: MLIR (Multi-Level Intermediate Representation). Do not build an IR from scratch. MLIR is the industry standard (used by Apple, Google, AMD) for building custom AI compilers. You will build a custom MLIR dialect for TPT-IR.
Math/Graph Engine: Apache TVM (for initial operator fusion and graph optimization).
Format Adapters: gguf-py (GGUF), safetensors (SafeTensors), onnx (ONNX), tensorflow (TF SavedModel + TFLite), flax + orbax (JAX/Flax checkpoints), auto-gptq / autoawq (GPTQ/AWQ metadata parsing).
4.2 FPGA Module (TPT Fusion)
Language: Python + Amaranth HDL. Amaranth is a Python-based hardware description language. It allows you to write Python code that generates Verilog, making the auto-routing of HBM controllers vastly easier to program.
Synthesis & Place/Route: Yosys (Open-source logic synthesis) and Nextpnr (Open-source place and route). We will wrap these tools so the user doesn't have to interact with the command line.
Memory Controllers: Integration with LiteX (specifically LiteDRAM for HBM interfacing).
4.3 Analog Module (TPT Element)
Simulation Engine: Xyce or ngspice. These are the premier open-source SPICE simulators. Xyce is developed by Sandia National Labs and handles large, complex analog circuits much faster than ngspice.
Python Wrapper: PySpice to control the simulation engine via Python.
The "Reality Check" ML Engine: PyTorch. We will train a lightweight neural network on thousands of SPICE simulation runs to predict thermal/noise drift instantly, rather than running slow, brute-force SPICE simulations for every error check.
4.4 Swarm Module (TPT Alloy)
Graph Partitioning: METIS or KaFFPa (C++ libraries). These are the gold standard for dividing massive graphs (neural networks) into smaller sub-graphs (microcontroller nodes) while minimizing the "cut" (data traveling between nodes).
Firmware Generation: Rust. Rust is perfect for generating highly optimized, memory-safe C/C++ firmware for microcontrollers on the fly.
Hardware Abstraction: PlatformIO or Zephyr RTOS integration, so the generated firmware can easily target anything from an ESP32 to a custom RISC-V chip.
4.5 UI/UX & Dashboard (TPT Observer)
Frontend: React.js + Next.js.
3D Visualization: Three.js or React Three Fiber. Crucial for visually rendering the physical topology of the Swarm (nodes and wires) and the PCB layouts for the Analog module.
Backend API: Go (Golang). Go is incredibly fast, handles concurrent connections beautifully, and is perfect for streaming real-time telemetry data from the hardware to the dashboard via WebSockets.
Styling: Tailwind CSS with a custom "industrial blueprint" dark-mode theme (dark grays, neon cyan/amber accents, monospaced fonts for data).
4.6 Automation Philosophy
TPT Crucible treats manual steps as bugs. Every step that requires a human to wait, decide based on opaque output, or retry after a failed tool invocation is a candidate for automation. Specific principles:
- Toolchain errors are always translated to plain English before reaching the user. Raw Yosys/Nextpnr/Xyce stderr is intercepted, matched against a catalog of ~30 common failure patterns, and surfaced as structured messages with suggested actions. When a failure is novel (not in the catalog) and an LLM provider is configured, the full error context is sent to the LLM and the response appears in Observer as an "AI Diagnosis" panel.
- Quantization is decided by the compiler, not the user. The auto-search loop (section 3.1) removes the trial-and-error cycle.
- Compilation results are always regression-checked. On every `tpt-catalyst pack` invocation, if a prior `.tptpkg` exists for the same model+target, the validator runs automatically in the background and diffs accuracy before the user flashes new firmware.
- Flashing is always parallel. For WiFi-capable nodes (ESP32), OTA flash broadcasts to all nodes simultaneously. For USB, multiple connected hubs are utilized in parallel.
- Communication parameters are auto-tuned. After partitioning, a SiL-based latency sweep over WiFi message size, batch size, and retry parameters selects the configuration that minimizes p99 inter-node latency within the firmware memory budget.
- AI assistance degrades gracefully. All AI-powered features (LLM error diagnosis, AI-generated validation prompts, mixed-precision quantization search, synthesis constraint auto-tuner) are hidden when no LLM provider or trained ML model is configured. No degraded partial experience.

4.7 Hardware-in-the-Loop Training (tpt-train extension)
Beyond recording activation profiles, tpt-train supports a closed-loop training mode. After deploying a model to real hardware or SiL, the validator collects per-layer output deviations between hardware and a CPU reference. These deviation profiles are fed back to a `TPTHardwareAwareCallback` as a regularization signal — the model is fine-tuned to be robust to the specific error characteristics of its target hardware (quantization rounding, memory bandwidth bottlenecks, analog component drift). This is an optional training phase that runs after initial model training and before final compilation.

4.7 Accessibility & Democratization
Zero-Install Browser Demo. The Rust crates (tpt-catalyst, tpt-alloy) compile to WebAssembly and run entirely in the browser via a Web Worker. The SiL emulator drives the Observer frontend with no installation required. A hosted demo at a public URL lets anyone compile TinyLlama and watch the swarm visualization in under 60 seconds.
Pre-compiled Package Marketplace. A community registry of `.tptpkg` files covering popular model + hardware combinations (e.g., `tpt get tinyllama-q4-16xesp32`). Packages are versioned, SHA-256 verified, and include accuracy delta metadata vs. CPU baseline. For common configurations, this eliminates the compilation step entirely.
Open Hardware Reference Board. A KiCad PCB design for an 8- or 16-node ESP32 swarm carrier board: standardized bus connectors between nodes, shared 5V power rail, UART programming headers, per-node status LEDs, and a JLCPCB-compatible BOM targeting under $15 assembled. Stored in `hardware/reference-designs/alloy-carrier/`. This removes the physical breadboard barrier for first-time builders.
One-Line Bootstrap. `pip install tpt-crucible` installs the full toolchain with optional extras: `pip install tpt-crucible[fpga]` adds Yosys/Nextpnr wrappers, `pip install tpt-crucible[swarm]` adds PlatformIO. Without optional extras, the wizard automatically falls back to SiL emulation so the user can explore the full workflow without installing any synthesis toolchains.

5. Development Roadmap
Phase 1: The Catalyst & The Swarm (Months 1-6)
Build TPT Catalyst (MLIR integration) to parse PyTorch models.
Build TPT Alloy (Swarm module). This is the easiest to demo.
Milestone: A user can load a small LLM (like TinyLlama) into TPT Catalyst, partition it via TPT Alloy, and flash it to 16 ESP32s to run a basic chat interface.
Phase 2: The Silicon Canvas (Months 6-12)
Build TPT Fusion (FPGA module). Integrate Amaranth HDL and LiteX.
Milestone: A user can select a Xilinx Alveo FPGA board in the UI, and TPT Fusion outputs a ready-to-flash bitstream that runs a quantized AI model using the board's HBM.
Phase 3: The Physics Engine (Year 2)
Build TPT Element (Analog module). This requires the most deep-tech R&D. Train the ML "Reality Check" model on SPICE data.
Milestone: A user designs a simple 3-layer analog neural network, TPT Element simulates the thermal drift, and outputs a KiCad PCB file ready for manufacturing.
Phase 4: The Observer (Year 2+)
Build the TPT Observer dashboard to unify telemetry across all three hardware types.
6. Key Differentiators (The "TPT" Advantage)
We don't compete with Nvidia on digital GPUs. We completely ignore the traditional GPU path.
We embrace physical constraints. While commercial tools try to hide hardware limitations, TPT Crucible exposes them (via the Analog Reality Check and Swarm Topology mapper) and uses software to engineer around them.
Open-Core Philosophy. The underlying compilers (MLIR, Yosys, Xyce) are open-source. TPT Crucible provides the "glue," the UI, and the proprietary AI-driven optimization layers, allowing the community to contribute hardware drivers.
FPGA Overlay Architecture. Pre-synthesized configurable datapaths cut per-model FPGA compile time from hours to seconds, making iterative development practical for the first time.
Transformer-Native Swarm Partitioning. Attention-head parallel + layer-serial hybrid partitioning is designed specifically for the dominant model architecture (transformers), not just generic layer-wise graph cuts.
KV Cache-Aware Memory Planning. The partition planner allocates KV cache across nodes alongside compute — without this, autoregressive inference silently OOMs on memory-constrained swarm nodes.
Fault-Tolerant Inference. Swarms degrade gracefully when nodes fail; inference continues rather than hanging. This is a prerequisite for production deployment of swarm hardware.
Zero-Friction Entry. Browser-based demo, pre-compiled package marketplace, and a reference carrier board collectively remove every major barrier (install friction, hardware wiring, compilation time) that has historically gated access to custom hardware AI deployment.
Universal Model Ingestion. SafeTensors, HuggingFace Hub, TFLite, AWQ/GPTQ, EXL2, JAX/Flax, and Llamafile alongside the original GGUF/ONNX/PyTorch/TF inputs. If the model exists in the wild, Catalyst can read it.
Automation-First Pipeline. Every step that requires manual iteration (quantization selection, pre-flight review, accuracy regression, firmware flashing, communication tuning) is automated by default. The user sets intent (accuracy budget, target hardware) and the pipeline executes without waiting for human input at each stage.
AI-Augmented Toolchain. LLM error diagnosis, mixed-precision quantization search, synthesis parameter auto-tuning, and domain-aware validation prompt generation are layered on top of the deterministic pipeline — all opt-in, all degrading gracefully when no provider is configured.
